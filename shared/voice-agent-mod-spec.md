# Minecraft Mod 仕様書（音声入力エージェント / Jarvis）

## 1. 目的

Minecraft Java 1.21.1 向けに、常時待機型の音声入力エージェントを提供する Mod を開発する。  
プレイヤーが「Hey ジャービス」と発話すると、右上に Jarvis UI が表示され、以後の会話を解釈して返答・コマンド実行補助を行う。

## 2. スコープ

### 2.1 対象

- 音声の常時入力（待機）
- ウェイクワード「Hey ジャービス」の検出
- Jarvis UI の表示/非表示
- 会話応答（MVP はテキスト表示のみ。音声合成は将来拡張）
- 条件付きで Minecraft コマンドを自動入力・実行

### 2.2 非対象（初版）

- マルチ言語対応（初版は日本語優先）
- 高度な NPC 制御や外部サービス連携の拡張機能
- 専用サーバープラグインとの密結合

### 2.3 技術方針（確定）

- Mod ローダーは `Fabric` を採用する
- ウェイクワード受付（「Hey ジャービス」検出）はローカル処理のみで行う
- ウェイクワードエンジンは MVP では `Porcupine` を採用する（非商用運用では Free プランを利用）
- Jarvis 起動後の音声認識（STT）と応答生成は API Key を使った外部 API 呼び出しを既定とする
- AWAKE 時の音声送信は Realtime API のストリーミングを優先し、失敗時は音声ファイル一括送信へフォールバックする
- モデル既定値は `STT=gpt-4o-mini-transcribe`、`応答生成=gpt-5-mini` とする

## 3. ユースケース

1. プレイヤーが通常プレイ中に「Hey ジャービス」と話す
2. 右上に Jarvis UI が表示され、待受状態に遷移
3. プレイヤーが「今の座標を教えて」「朝にして」などの指示を話す
4. Jarvis が内容を解釈して回答し、必要なら `/time set day` などのコマンドを提案または実行する

## 4. 機能要件

### FR-01 常時音声待機

- Mod 起動中はマイク入力を継続的に受け付ける
- 常時待機は軽量モードで動作し、CPU 負荷を抑える
- マイク未接続時は UI に警告を表示

### FR-02 ウェイクワード検出

- キーフレーズ: 「Hey ジャービス」
- 検出成功時、Jarvis UI を右上に表示しセッション開始
- 誤検出を減らすため、一定しきい値以上の信頼度で起動する
- `IDLE` 状態では API を呼ばない（課金ゼロで常時待機）

### FR-03 Jarvis 表示中の対話

- Jarvis UI 表示中のみ自由会話を受け付ける
- 返答は最低限テキスト表示する
- VAD（無音検出）を採用し、`-45 dB` 以下が `1.0 秒` 継続した場合を無音区間として扱う
- 無音区間が累計 `10 秒` 継続した場合、会話タイムアウトとして `IDLE` へ戻る
- API 呼び出しは `AWAKE` 状態での対話時のみ行う
- AWAKE 時の STT は API 優先（既定）とし、音声チャンクのストリーミング送信を基本経路にする

### FR-04 コマンド自動実行

- 発話内容が Minecraft コマンド実行意図と判定された場合、コマンド生成を行う
- 実行モードを2段階で提供する
  - 提案モード（既定）: コマンド候補を表示し、ユーザー承認後に実行
  - 自動モード: 条件一致時に即時実行
- 危険コマンド（例: 大規模破壊、権限変更）は確認必須

### FR-05 状態管理

- エージェント状態:
  - `IDLE`（常時待機）
  - `AWAKE`（Jarvis 表示・会話受付）
  - `EXEC_CONFIRM`（コマンド確認）
  - `COOLDOWN`（連続誤起動防止）
- 状態遷移はログに記録する
- `COOLDOWN` は 3 秒固定（MVP 既定）
- `COOLDOWN` 中はウェイクワード検出を無効化する
- 3 秒経過で自動解除し `IDLE` へ戻る
- `IDLE` 復帰後にウェイクワード再検出で `AWAKE` へ遷移する

## 5. 非機能要件

### NFR-01 パフォーマンス

- 平常時（`IDLE`）CPU 使用率は実用範囲で安定動作すること（MVP では数値目標を設けない）
- 音声応答の体感遅延は実用上許容できる範囲に収める（MVP では数値目標を設けない）

### NFR-02 安全性

- コマンド実行時は実行者・内容・時刻をローカルログ保存
- 危険操作に対しては二重確認を可能にする

### NFR-03 可用性

- 音声認識が失敗してもゲーム本体が継続可能
- 外部音声エンジン障害時はテキスト通知して機能を縮退

### NFR-04 プライバシー

- 初版は「ローカル処理優先」を基本方針とする
- 外部 API 利用時は明示的に設定で有効化し、送信可否を説明する

## 6. UI 仕様（初版）

- 位置: 画面右上固定
- 見た目: アイアンマン Jarvis 風（円弧/ホログラム調）
- 表示要素:
  - 状態インジケータ（Listening / Thinking / Executing）
  - 最新認識テキスト
  - 最新応答テキスト
  - コマンド提案カード（承認/キャンセル）

### 6.1 UI 競合時の優先制御

- 優先順位は「システム GUI > 操作系 GUI（チャット/インベントリ等）> Jarvis UI」とする
- チャット入力欄表示中は Jarvis の音声受付を一時停止し、チャット終了時に再開する
- インベントリ表示中は Jarvis の自動実行を禁止し、提案表示のみ許可する
- キーボード入力はフォーカス中 GUI のみ受け付け、Jarvis UI は非フォーカスオーバーレイとして動作する
- GUI 競合中に危険コマンドは実行せず保留し、競合解消後に確認フローへ戻す

## 7. コマンド実行ポリシー

- 実行権限プリセットを提供する（`Normal` / `Unlimited`）
- `Normal`（既定）:
  - 自動実行を許可するコマンドは低リスク系のみ（例: `/time`, `/weather`, 一部 `/gamerule`）
  - それ以外は提案のみ（ユーザー承認後に実行）
- `Unlimited`:
  - ユーザーが明示的に有効化した場合のみ利用可能
  - `/kill` と `/tp` は自動実行を許可する
  - ブロックリストを除き、提案なし自動実行を許可する
- 常時ブロック（プリセット共通）: `/op`, `/deop`, `/ban` などの権限管理系コマンド
- 非許可コマンドは提案のみで自動実行しない

### 7.1 シングル/マルチプレイ時の権限境界

- シングルプレイでは提案モード/自動モードの両方を利用可能とする
- マルチプレイの非 OP プレイヤーは提案モードのみ利用可能とし、自動実行は無効化する
- マルチプレイの OP プレイヤーは設定により自動実行を有効化できる
- 危険コマンドの判定と確認要否は 7.2 のルールに従う
- 実行前にプレイヤー権限で実行可能かを必ず検証し、権限不足時は未実行で理由を表示して提案モードへフォールバックする

### 7.2 危険コマンド判定ルール（MVP）

- 一次判定はコマンド種別、二次判定は引数（対象範囲・対象数）、最終判定は実行権限で行う
- `/fill` と `/clone` は対象体積（`x*y*z`）がしきい値 `32768` を超える場合は常時ブロックする（`Normal` / `Unlimited` 共通）
- `/fill` と `/clone` がしきい値以下の場合は `Normal` では提案のみ、`Unlimited` では自動実行を許可する
- `/kill` は `Unlimited` で自動実行を許可する（`Normal` は提案のみ）
- `/tp` は `Unlimited` で自動実行を許可する（`Normal` は提案のみ）
- 権限不足時は常に未実行とし、理由を表示して提案モードへフォールバックする

## 8.1 API Key 取り扱い要件

- API Key はユーザーが mod 設定画面で入力できる
- 保存時は平文を避け、暗号化ファイルに保存する（MVP では OS 資格情報ストア連携は行わない）
- ログ/クラッシュレポートへ API Key を出力しない
- キー未設定時はローカル機能のみ有効化し、API 機能は無効化する
- MVP では API Key の自動ローテーションは実施しない（ユーザーが設定画面で手動更新する）

## 8.2 API 利用上限超過時の仕様

### 前提条件

- API 利用量の要求値を `requestedValue`、設定上限を `limit` とする
- `requestedValue <= limit` の場合のみ通常処理を実行する
- `requestedValue > limit` の場合は上限超過として扱う

### 例外仕様（上限超過）

- 上限超過を検知した時点で処理を即時中断する
- エラーコード `LIMIT_EXCEEDED` を返却する
- エラーメッセージは以下で統一する  
  `上限を超過しているため処理を実行できません。入力値を調整して再実行してください。`
- 上限超過時は副作用を発生させない（DB更新・外部 API 連携・通知送信を行わない）
- エラーログには対象 ID / `requestedValue` / `limit` / 発生日時 / トレース ID（存在する場合）を記録する

### API 仕様

- 判定条件:
  - `requestedValue <= limit` は正常処理
  - `requestedValue > limit` はエラー終了
- エラーレスポンス例:

```json
{
  "errorCode": "LIMIT_EXCEEDED",
  "message": "上限を超過しているため処理を実行できません。入力値を調整して再実行してください。"
}
```

- HTTP ステータス（REST の場合）は `400 Bad Request` を推奨し、業務ルール違反として扱う実装では `422 Unprocessable Entity` も許容する
- クライアントは `LIMIT_EXCEEDED` 受領時に自動再試行せず、入力値調整後に再実行する

## 8.3 ログ保存・保持・削除ポリシー

- ログ保存先は `/.minecraft/logs/jarvis/` とする
- ログファイルは `jarvis-YYYY-MM-DD.log` 形式で日次ローテーションする
- 既定の保持期間は `30 日` とし、設定で `7 / 30 / 90 / 無制限` を選択可能とする
- クライアント起動時に保持期限を超過したログを自動削除する
- ログ総容量が `100 MB` を超過した場合は、古いファイルから順に削除する（容量上限を優先）
- 設定画面に「ログを今すぐ削除」を提供し、ユーザーが即時削除できるようにする
- API Key および音声生データはログへ記録しない

## 9. 開発フェーズ（MVP）

1. 常時待機 + ウェイクワード検出
2. 右上 Jarvis UI 表示
3. 表示中の音声認識とテキスト返答
4. コマンド提案モード
5. 自動実行モードと安全制御

### 9.1 IDLE 判定開発戦略（確定）

- IDLE 判定は Mod 側（Java）で実装し、Jarvis CLI へは依存しない
- 判定コアは Minecraft/Fabric 依存を排除した純 Java モジュールとして実装する
  - `WakeWordDetector`（ウェイクワード判定）
  - `VadGate`（無音判定）
  - `IdleStateMachine`（`IDLE -> AWAKE -> COOLDOWN -> IDLE`）
- 音声入力はアダプタで差し替え可能にする
  - 実行時: `MicInput`
  - テスト時: `WavInput`
- 単体テストは Mod 起動なしで実行可能とし、WAV 注入で以下を検証する
  - `wake.wav` で `IDLE -> AWAKE` に遷移する
  - `noise.wav` で遷移しない
  - `COOLDOWN` 中は再検知しない
  - しきい値境界で誤遷移しない
- 統合テストは Mod-コア間の接続確認に限定し、実機マイク確認は手動スモークテストで実施する
- 受け入れ条件として、IDLE 判定の自動テスト全通と、実マイクでの 1 回以上のウェイク確認を必須とする

### 9.2 テスト責務分離（確定）

- AI は機械可読な自動テスト（unit / contract / e2e / bridge）を担当する
- 人間は実機操作を伴う手動テスト（実マイク、UI 体感、LLM 応答品質評価）を担当する
- 自動テスト結果は機械可読形式（JUnit XML / JSON サマリ）で保存する
- リリース判定は「自動テスト全通 + 手動スモーク完了」の両方を満たすこと

## 10. 受け入れ基準（初版）

- 「Hey ジャービス」で 2 秒以内に UI 表示される
- Jarvis 表示中の発話に対し、テキスト応答が返る
- 指示に応じてコマンド候補が生成される
- 提案モードで承認後、コマンドが正しく実行される
- 危険コマンドは承認なしで実行されない
